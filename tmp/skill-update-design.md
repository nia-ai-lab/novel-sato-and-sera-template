# スキル変更設計書

作成日: 2026-03-15
対象: `soudoku-novel-builder` と関連する画像生成 / Web ビルド資産

## 1. 依頼意図の理解

現時点での理解は次のとおりです。

- Web UI に直接出す画像、特に `節画像` の基準を、現行の縦長中心から `16:9 / 1920x1080` 中心へ切り替えたい。
- コンパイル後の Web UI は、`スマホ縦` と `横画面（PC / スマホ横）` で別レイアウトにしたい。
- `PC / スマホ横` では、1枚の `1920x1080` 画像をトリミングせずフル表示し、その上で本文も読みやすく上質に見せたい。
- `スマホ縦` では、1シーンを縦に読み進める間、背面の横長画像が `左 -> 右` にパンする読書体験にしたい。
- `キャラクター画像` は Web UI へ直接出さないため、今まで通りのサイズのまま維持したい。
- `背景画像` は大量生成しない。各話ごとに、`サトーパート` を基準にした背景画像を 1 枚、標準構成では計 6 枚だけ生成したい。
- その `背景画像を作った prompt` を、同じ話の `サトーパート` と `セラパート` の節画像生成へ流用したい。
- `背景画像そのもの` を節画像の参照画像に入れるのではなく、`背景 prompt を共有する` ことで、セラパートでは同じ舞台の裏側、別角度、別気配を表現しやすくしたい。
- 今回は実装ではなく、まずこの変更をどう入れるかの設計書を作り、レビュー承認後に実装へ進みたい。

この設計案では、画像の扱いを次のように分ける。

- タイトル画像: 今回の横長化対象
- 節画像: 今回の横長化対象
- 背景画像: 各話ごとの舞台基準 prompt を作る用途で維持
- キャラクター画像: サイズ変更対象から外し、参照素材として現行維持

## 2. 現状整理

確認できた現状は次のとおり。

- 画像生成スクリプトは `9:16` 前提。
- prompt テンプレート既定値も `1080x1920` または `1440x2304` の縦長前提。
- `scripts/build-web-novel.mjs` は `soudoku-novel-builder/assets/mobile-reader` を配信用テンプレートとして使っている。
- 現在の Web UI は `mobile-reader` 名の通り、実質的にスマホ縦スクロール体験が中心。
- `780px` 以上では疑似端末フレームに入るだけで、`PC で横長画像をフル表示する` レイアウトにはなっていない。
- 節画像 prompt も `vertical smartphone-friendly` と書かれており、新要件と逆向き。
- 背景画像生成の仕組みはあるが、節画像生成時に `background-concepts.json` や背景 prompt は自動参照されていない。
- 節画像生成時に参照されている画像は、現在はキャラクター画像だけである。

## 3. 提案する変更方針

## 3.1 画像仕様の対象を分ける

今回の変更では、全画像を一律にはしない。

- `節画像` は `16:9 / 1920x1080` に変更する
- `タイトル画像` も `16:9 / 1920x1080` に変更する
- `キャラクター画像` は現行サイズを維持する
- `背景画像` は各話の舞台基準用として維持し、`16:9 / 1920x1080` に変更する

理由:

- キャラクター画像は Web UI の表示対象ではなく、節画像用の参照素材だから。
- 背景画像は読者向け UI の主要表示物ではなく、`各話の舞台・空気・光` を整理する制作資産として位置づけ直すから。

## 3.2 節画像と背景画像の関係を再設計する

変更対象:

- `scripts/generate-title-image.mjs`
- `scripts/generate-background-concept.mjs`
- `scripts/generate-episode-image.mjs`
- `scripts/init-novel-project.mjs`
- `scripts/build-episode-image-manifest.mjs`
- `soudoku-novel-builder/references/prompt-files.md`

変更内容:

- 背景画像は `各話ごとに 1 枚` を基本とする。
- 標準構成 `3章 / 6話 / 12節` では、背景画像は `6 枚` を標準生成数とする。
- 各背景画像は、その話の `サトーパート` を基準に prompt を作る。
- 節画像生成時は、`背景画像そのもの` を参照画像として使うのではなく、`その背景画像を作った prompt` を流用する。
- 同じ話に属する `サトーパート` と `セラパート` は、同じ背景 prompt を共有する。
- 節画像 prompt には、共有背景 prompt に加えて、その節固有の本文要約、beats、キャラクター参照、scene override を加える。
- scene prompt を `縦長スマホ向け` から `横長 16:9 のシネマティック画面` へ変更する。
- 縦スマホ表示時のパン表示を考慮し、節画像 prompt に `主要被写体は中央寄り`、`左右端に重要情報を置きすぎない`、`パンしても破綻しない余白構成` を追記する。

設計意図:

- サトーパートで見えている舞台を、その話全体の `正面側の舞台基準` として固定する。
- セラパートでも同じ背景 prompt を共有することで、同じ場所の裏側、死角、監視側、別角度を自然に派生させやすくする。
- 背景画像の枚数を話数単位に抑えることで、生成時間と運用負荷を増やしすぎない。

理由:

- 画像仕様を混在させると、Web 側の見せ分けが複雑になる。
- `横画面でフル表示` と `縦画面で横パン` の両方を成立させるには、元画像が同じ横長仕様である方が扱いやすい。
- 背景画像を `話単位の舞台プロンプト資産` として使うと、節画像との整合性を高めつつ、生成枚数を抑えられる。

## 3.3 コンパイル Web UI を 2 モード化する

方針:

- `スマホ縦モード`
- `横画面モード（PC / スマホ横）`

の 2 系統に分ける。

実装上は `orientation` と `viewport` を使って切り替える。

### A. スマホ縦モード

読書体験:

- 現行の `縦スクロールで読み進める` 基本体験は維持する。
- 各シーンで、画像は `sticky` な背面レイヤとして見せる。
- 本文カードを縦に読み進める間、そのシーン画像は `左 -> 右` にパンする。
- パン量は `シーン内スクロール進捗` に連動させる。小説全体のスクロール量ではなく、各シーンごとに 0 から 1 を計算する。

見た目:

- 画像は縦画面全体を覆う `cover` 表示にする。
- 横長 16:9 画像を縦画面へ入れるため、左右に余剰が出る。その余剰をパンに使う。
- 本文はガラス調または半透明カードで読みやすく重ねる。
- 現在の雰囲気は継承しつつ、本文カードの幅、行間、字間、余白を読みやすく再調整する。

実装メモ:

- `app.js` に `scene local progress` の計算を追加する。
- 各 `.scene` に CSS 変数 `--pan-progress` または `--pan-x` を設定する。
- `transform: translate3d(...)` を使い、`requestAnimationFrame` 内で更新してスクロール性能を守る。

### B. 横画面モード（PC / スマホ横）

読書体験:

- `1920x1080` 画像をトリミングせず、シーンの主舞台としてフル表示する。
- 本文は画像の上に重なる読みやすいパネルとして置く。
- 横方向の広さを使って、本文、章話情報、進捗、目次の配置に余裕を持たせる。
- 操作はスマホ縦モードと同じく、`縦スクロールで読み進める` 方式を維持する。
- 横画面モードでも、シーン遷移や本文読書の主操作を横スワイプやページ送りにはしない。

見た目:

- 疑似スマホフレームは廃止する。
- 画面全体を 16:9 ステージとして使う。
- シーン画像は `object-fit: contain` を基本にし、全体が見切れないようにする。
- 余白が出る環境では、背景ぼかしやトーン付きマットで埋めて没入感を維持する。
- 本文パネルは片側寄せの半透明面として置く。PC では幅を広め、スマホ横では少し薄くコンパクトにする。

重要な設計判断:

- `16:9 画像をフル表示する` と `本文を横に別置きする` は、ちょうど 16:9 の端末では両立しない。
- そのため、横画面では `画像はフル表示` を優先し、本文は `画像の上に重ねる` 方式を推奨する。
- より横幅の広い PC では、画像外側の余白にナビゲーションや補助情報を逃がす。

## 3.4 Web テンプレートを `mobile-reader` から `responsive-reader` へ再定義する

提案:

- テンプレート資産の役割が `モバイル専用` ではなくなるため、名称も更新する。

想定変更:

- `soudoku-novel-builder/assets/mobile-reader/` を `soudoku-novel-builder/assets/responsive-reader/` へ移行
- `scripts/build-web-novel.mjs` のテンプレート参照先も更新

理由:

- 実態と名前が一致した方が、スキルの保守性が上がる。

最小変更を優先する場合の代替案:

- ディレクトリ名は残し、中身だけ responsive 化する。

この設計では、可読性を優先して `responsive-reader` への名称更新を推奨する。

## 3.5 story-data と背景 prompt 連携の構造

現行の `docs/story-data.js` は、章・話・節・beats を既に十分持っている。

方針:

- `scripts/build-web-novel.mjs` のデータ構造は極力維持する。
- UI 側のレイアウト分岐やパン演出は、主に `app.js` と `styles.css` 側で吸収する。
- ただし節画像 manifest 側には、`話単位の背景 prompt` を合成するための入力を追加する。

追加する設計:

- `prompts/background-concepts.json` を `話単位` で定義する
- 各背景 entry に、対応する `talkKey` または `talkNumber` を持たせる
- `build-episode-image-manifest.mjs` で、各節が属する `話` に対応する背景 prompt を引く
- その背景 prompt を、各 `episode.prompt` の前段コンテキストとして合成する

想定する合成順:

1. global scene prompt
2. 該当話の背景 prompt
3. 節固有の scene summary / beats
4. キャラクター参照情報
5. scene override / promptSuffix

補足:

- ここで流用するのは `背景画像そのもの` ではなく `背景画像生成 prompt` である。
- そのため `generate-episode-image.mjs` で新たに背景画像ファイルを添付する実装は不要。

## 4. 具体的な実装差分案

## 4.1 画像生成系

- `generate-title-image.mjs`
  - `aspectRatio: "16:9"`
  - `fixedWidth/fixedHeight = 1920/1080`
- `generate-background-concept.mjs`
  - `aspectRatio: "16:9"`
  - `fixedWidth/fixedHeight = 1920/1080`
  - 話単位背景の生成を前提に、prompt 定義例を更新
- `generate-episode-image.mjs`
  - 同上
  - safe area 指示を強化
  - 背景画像ファイルの参照追加は不要
- `generate-character-portrait.mjs`
  - 画像サイズは変更しない
- `build-episode-image-manifest.mjs`
  - `background-concepts.json` を読む
  - 話単位背景 prompt を節画像 prompt に合成する
  - prompt 文言を `vertical smartphone-friendly` から新仕様へ変更
- `init-novel-project.mjs`
  - 背景 prompt テンプレートを `各話 = 1背景` 前提に更新
  - 節画像 spec を 1920x1080 に変更
  - キャラクター画像 spec は据え置き

## 4.2 Web ビルド系

- `build-web-novel.mjs`
  - テンプレート参照先の更新
  - description 文言を `縦スクロール Web 小説` から `レスポンシブな画像付き Web 小説` へ調整
  - story data は極力据え置き

## 4.3 フロントエンド資産

- `index.html`
  - 横画面用のレイアウトコンテナを追加
  - 目次 / 進捗 / 本文パネルの構造を responsive 化
- `styles.css`
  - モード別レイアウトを全面的に再設計
  - 縦画面では sticky + pan 前提
  - 横画面では full-stage + overlay panel 前提
- `app.js`
  - シーンごとのパン進捗計算
  - モード切替時のレイアウト再計算
  - 横画面モードでも縦スクロール基準の現在位置計算を維持
  - 目次・現在位置追従は維持

## 5. 受け入れ条件

承認後の実装完了条件は次のとおり。

- 新規プロジェクト作成時、`節画像` と `背景画像` の spec が `1920x1080` になる。
- 新規プロジェクト作成時、`キャラクター画像` の spec は従来サイズのまま維持される。
- 標準構成では、背景画像が `各話ごとに 1 枚`, 計 `6 枚` 生成できる設計になる。
- `build-episode-image-manifest.mjs` が、各節に対して同じ話の背景 prompt を取り込める。
- サトーパートとセラパートの節画像が、同じ話の背景 prompt を共有できる。
- 既存の画像生成スクリプトが `節画像` と `背景画像` では `16:9` で生成し、最終出力も `1920x1080` になる。
- `PC / スマホ横` では節画像をフル表示し、本文も重ね表示で十分読める。
- `PC / スマホ横` でも、読書操作は縦スクロールで一貫している。
- `スマホ縦` では、本文を縦スクロールするとシーン画像が `左 -> 右` にパンする。
- 章メニュー、進捗表示、タイトルページが新レイアウトで破綻しない。
- スキル文書と README が新仕様を説明している。

## 6. リスクと対策

- スマホ縦のパン演出が重くなる
  - `requestAnimationFrame` と CSS transform のみで実装する
- 横画面で本文と画像が干渉する
  - 本文面に最大幅、背景ぼかし、濃度調整を入れる
- 既存 `mobile-reader` 前提の命名がズレる
  - `responsive-reader` へ名称更新する
- 背景 prompt をそのまま節画像へ流すと、人物不要指示が衝突する
  - 背景 prompt は `舞台・時間・光・空気感` の継承に使い、人物排除系の指示は節画像用に調整する
- 話単位背景と節固有描写がズレる可能性がある
  - 背景 prompt はベースにとどめ、節固有の summary / beats / override を後段で必ず重ねる

## 7. レビュー時に見てほしい点

- 横画面で本文を `画像上の重ねパネル` にする案でよいか
- スマホ縦のパン方向を全シーン一律 `左 -> 右` にする案でよいか
- テンプレート名を `responsive-reader` へ変更する案でよいか
- 背景画像を `各話につき 1 枚 / 計 6 枚` に固定する案でよいか
- サトーパート基準の背景 prompt を、同じ話のセラパートにも流用する案でよいか

## 8. 実装順序案

1. `背景画像 = 各話 1 枚` の prompt 設計へ変更
2. `build-episode-image-manifest.mjs` で背景 prompt 流用を実装
3. `節画像` と `背景画像` の spec を 16:9 に変更
4. responsive-reader テンプレートへ再構成
5. app.js に縦画面パンと横画面 full-stage を実装
6. SKILL.md / README を更新
7. サンプルビルドで表示確認
